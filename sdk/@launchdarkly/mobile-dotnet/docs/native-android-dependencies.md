# Native Android dependencies (audit)

This document describes what ships under `runtimes/android/native/` in the fat NuGet package and how it is wired. It supports ongoing work to **reduce consumer DEX/R8 cost** (fewer redundant bytecode inputs).

## Source of truth

- Item definitions: [`observability/NativeAndroidDeps.props`](../observability/NativeAndroidDeps.props) (explicit `_LDNativeJAR` **Remove** rules; plus [`Pack.Generate.targets`](../observability/Pack.Generate.targets) target `StripExcludedOtelJarsFromPack` removes the same JARs from `None` before `GenerateNuspec`, because multi-TFM `dotnet pack` can still list excluded files)
- Consumer MSBuild wiring: `buildTransitive/LaunchDarkly.SessionReplay.targets` (generated by `_GenerateConsumerBuildTransitiveCore` in [`Pack.Generate.targets`](../observability/Pack.Generate.targets), checked in for review)
- Manifest used for pack validation: `buildTransitive/native-manifest.txt`

## Artifact groups

| Group | Patterns | Role |
|--------|----------|------|
| OpenTelemetry Android agents | `android-agent-*.aar`, `android-instrumentation-*.aar`, `core-*-alpha.aar`, `common-*.aar`, `crash-*`, `anr-*`, `activity-*`, `fragment-*`, `network-*`, `services-*`, `session-*`, `slowrendering-*`, `startup-*` | OTel Android RUM / instrumentation |
| LaunchDarkly Android | `launchdarkly-android-client-sdk-*.aar`, `launchdarkly-*.jar` | LD mobile client |
| App bridge | `LDObserve-release.aar` (from `outputs/aar/`), `lib-release.aar` | Observability bridge + Kotlin library |
| Logging | `timber-*.aar` | Timber |
| OpenTelemetry Java API/SDK | `opentelemetry-*.jar` (with explicit excludes for autoconfigure / incubator test jars) | OTLP export and SDK pieces used from .NET integration |
| JSON / HTTP | `jackson-*.jar`, `okhttp-*.jar`, `okio-*.jar` | Serialization and HTTP used by bundled Java components |

## Overlap and duplicate-class risk

- **JARs alongside AARs:** Many AARs already bundle transitive Java libraries. Shipping **both** an AAR and separate JARs for the same logical library can inflate DEX merge work and occasionally cause **duplicate class** errors until one side is removed. The correct fix is decided **per version** by inspecting Gradle dependency trees and the contents of each AAR (e.g. `classes.jar` inside the AAR).
- **OpenTelemetry:** Multiple `opentelemetry-*.jar` lines are intentional for the exporter pipeline; trimming requires verifying nothing removes classes referenced at runtime.
- **Future consolidation (Gradle):** Where possible, embed helper JARs into a single library AAR or use a single “fat” Android artifact from upstream builds so the .NET package references **fewer** top-level `AndroidLibrary` / `AndroidJavaLibrary` items.

## Optional product split

Shipping **all** OTel Android agent AARs by default maximizes instrumentation coverage and **increases** compile-time and R8 work for every consumer. A possible future improvement is a **slim** package (core + bridge + minimal OTel) and an **add-on** package for extra agents—**product and API review required**.

## Regenerating consumer targets

After changing native outputs or globs in `NativeAndroidDeps.props`, regenerate checked-in files:

```bash
cd sdk/@launchdarkly/mobile-dotnet/observability
dotnet msbuild LDObservability.Fat.csproj -t:_GenerateConsumerBuildTransitiveCore -p:TargetFramework=net9.0-android -p:Configuration=Release
```

Or run the full fat pack script: [`scripts/pack_observability_fat.sh`](../scripts/pack_observability_fat.sh).
